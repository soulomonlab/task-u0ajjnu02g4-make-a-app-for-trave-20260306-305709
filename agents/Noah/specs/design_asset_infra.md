Title: Design Asset Hosting & Deployment Spec

Purpose
- Provide a repeatable, secure, and observable infrastructure for storing and serving design deliverables (PNG/SVG, exported Figma files, and canonical Figma links) so frontend and mobile teams can consume them reliably.

Goals
- Fast global delivery via CDN (CloudFront)
- Simple CI flow so designers or engineers can publish new assets by merging to main
- Clear naming, versioning, and metadata conventions to avoid asset drift
- Access control and secrets managed in GitHub Actions / AWS IAM
- Observability and runbooks for alerting and incident recovery

Requirements
- Public read access for static branding and UI assets served to web/mobile
- Private (signed URL) support option for pre-release or licensed assets
- Automated invalidation on major releases
- S3 bucket versioning enabled and lifecycle policy to expire old intermediate exports
- Audit logs (S3 access logs + CloudFront logs) exported to logging bucket

Design decisions (DevOps-owned)
1) Storage: AWS S3 + CloudFront for CDN in front. S3 bucket name pattern: <org>-design-assets-prod (use env var S3_BUCKET_NAME)
2) Deployment trigger: GitHub Action on merge to main that syncs output/design_assets/ to S3
3) Access control: IAM user for GitHub Actions with only s3:PutObject, s3:DeleteObject, s3:ListBucket on the target prefix and cloudfront:CreateInvalidation on the distribution. No long-lived credentials in repo.
4) Asset visibility: default public-read objects + cache-control headers. For private assets, implement a separate bucket or a prefix + signed URLs generated by backend.
5) Versioning & lifecycle: enable S3 versioning; lifecycle rule to move non-current versions to Glacier after 90 days and expire after 365 days (adjustable).
6) Observability: enable CloudFront standard logs + S3 server access logs. Create CloudWatch dashboard for 4xx/5xx errors, cache hit ratio, and requests per minute.

Naming & directory conventions
- Repository path: /design_assets/
- Subfolders:
  - /design_assets/figma_links/  -> .txt or .url files containing canonical Figma share links
  - /design_assets/wireframes/{home,search,itinerary,checkout}/v{1,2}/... -> PNG, SVG
  - /design_assets/components/{icons,logos}/v{X}/...
- File names: kebab-case with semantic suffixes, e.g., home-discovery-v1.png, booking-checkout-flow-v2.svg
- Metadata: attach object tags: project=travel-app, env=prod, designer=<name>

Cache policy
- UI static assets (icons, logos, SVGs): Cache-Control: public, max-age=31536000, immutable
- Page-level wireframes and PNG exports: Cache-Control: public, max-age=3600
- When publishing a major revision, bump the filename or version folder to avoid stale cache.

GitHub Secrets required (document in PRs when wiring pipeline)
- AWS_ACCESS_KEY_ID
- AWS_SECRET_ACCESS_KEY
- AWS_REGION
- S3_BUCKET_NAME
- CLOUDFRONT_DISTRIBUTION_ID

Monitoring & SLOs
- Availability target for asset CDN: 99.9% (SLO)
- Alerting:
  - CloudFront 5xx rate > 0.5% for 5m → P1
  - S3 403/AccessDenied errors spike → P0
- Runbooks: see section below

Runbook (high-level)
- P0: S3 access denied for deploys
  1. Check GitHub Actions run logs for AWS errors
  2. Verify IAM keys validity in Secrets and rotate if expired
  3. Confirm IAM permissions on the user (s3:PutObject, s3:ListBucket)
  4. Roll back to previous assets by restoring from S3 versioning or reverting PR
- P1: CloudFront serving stale assets
  1. Confirm new files exist in S3 with expected cache-control
  2. Create CloudFront invalidation for affected paths
  3. If widespread, trigger rollback by syncing previous commit from repo

Security notes
- Do NOT commit any AWS credentials. Use GitHub secrets.
- For private assets use signed URLs or a private prefix; coordinate with backend (Marcus) if signed URL generation is required.
- Rotate AWS keys every 90 days and record key ID usage in the audit log.
